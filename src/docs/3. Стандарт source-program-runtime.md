# Стандарт source → program → runtime

## Канонический pipeline

```text
persisted entity/source
        ↓ source compiler
normalized document / static IR
        ↓ EndgeCompiler
Endge.program / ProgramArtifact
        ↓ Endge.runtime
runtime host / live state
        ↓ adapter
Vue, Canvas или другой renderer
```

Каждый слой владеет только своим видом данных. Persisted entity хранит authoring source. Source compiler принимает TypeScript-подобный whitelist DSL и строит статический IR. `ProgramArtifact` является единственным compiled read-model для runtime. Runtime host владеет изменяемым состоянием, подписками и внешними ресурсами. Renderer только адаптирует host к UI.

## Source

Source — сохраняемое авторское описание, а не выполняемый JavaScript. Callback в DSL является синтаксической оболочкой статического expression tree. `eval`, `new Function`, import/require, произвольные вызовы, циклы, side effects и доступ к глобальному окружению запрещены.

```ts
body(({ prop }) => merge({ limit: 500 }, prop('filterPayload')))
```

Компилируется в IR:

```ts
{
  type: 'operation',
  operation: 'merge',
  arguments: [
    { type: 'object', properties: { limit: { type: 'literal', value: 500 } } },
    { type: 'read', source: 'prop', path: 'filterPayload' },
  ],
}
```

## ProgramArtifact

`Endge.program` хранит артефакты Filter, Query, DataView, Component и Composition. Payload содержит только materialized read-model. Source, diagnostics и dependencies остаются в общей оболочке `ProgramArtifact`, поэтому runtime не читает persisted entity напрямую и не компилирует source повторно.

Локальные source-first сущности компилируются в `ProgramArtifact.children`. Родительский payload хранит ссылки на child artifacts, а не их копии.

## Metadata

Source-first сущности `Query`, `DataView`, `Filter` и `Composition` могут объявить корневое поле `metadata`. Новый SFC-компонент объявляет metadata через `defineMetadata(...)`; его template-узлы — через compiler-only атрибут `:metadata`.

```ts
defineQuery({
  metadata: {
    'hub.tgo': {
      attributes: ['BestOn', 'FlightStatus'],
    },
  },
  // ...
})
```

Metadata:

- принимает только статический JSON-compatible object literal;
- группируется по пользовательским namespace, чтобы независимые consumers не делили плоское пространство ключей;
- не попадает в persisted backend-модель и transport/runtime payload;
- компилируется в единый `ProgramArtifact.metadata` с полями `self` и `nodes`;
- не выполняет функции и не читает runtime state — динамическая metadata является compile error.

Таким образом, Composition и другие consumers читают требования из compiled program, а не импортируют таблицу или конкретный UI-компонент.

## ValueExpression

Query, Composition и DataView используют один безопасный `ValueExpression` IR, один compiler и один runtime registry операций. Различаются только domain readers и orchestration-конструкции:

- Query: `prop(...)`, `response(...)`; выражения доступны в `request.body` и response-backed outputs;
- Composition: `prop(...)`, `fromOutput(...)`, `fromData(...)`, `fromStore(...)`, `fromFilter(...).fields(...)`, `metadata(...)`, `metadataOf(...)`;
- DataView: `path(...)` внутри `map(...)` и object `output`; `from`, legacy `join`, `map`, `spread`, `dataView` остаются структурными шагами pipeline DataView.

```ts
fromOutput('flights', 'rows')
  .where(and(eq(get('status'), 'active'), gt(get('delay'), 5)))
  .sortBy(get('std'))
  .map(pick(['id', 'status']))
```

`fromOutput('runtime')` собирает реактивный объект всех outputs runtime-ноды с их публичными именами. Linker фиксирует список имён внутри ProgramArtifact. `fromOutput('runtime', 'output')` читает один output без внешней объектной обёртки; явные вызовы можно использовать для ручной сборки собственной структуры.

Поддерживаемые immutable chain methods:

- доступ: `get`, `getOr`, `has`, `defaultTo`;
- объекты: `pick`, `omit`, `merge`, `defaults`, `compact`, `keys`, `values`, `entries`;
- массивы: `map`, `where`, `reject`, `find`, `some`, `every`, `flatMap`, `flatten`, `uniq`, `uniqBy`, `concat`, `take`, `drop`;
- порядок и индексы: `sortBy`, `groupBy`, `keyBy`;
- агрегаты: `size`, `sum`, `sumBy`, `min`, `max`, `minBy`, `maxBy`;
- строки: `trim`, `lowerCase`, `upperCase`, `split`, `join`.

Relational collection operations доступны во всех ValueExpression contexts:

- `leftJoin(left, right)` сохраняет все left records;
- `fullJoin(left, right)` сохраняет records с обеих сторон;
- `.by(path)` сопоставляет один одинаковый path или composite keys;
- `.by({ left, right })` сопоставляет разные paths;
- `.byAny(...)` объединяет альтернативные matching keys через OR;
- `.coalesce({ prefer?: 'left' | 'right' })` заполняет отсутствующие поля joined records.

`null` и `undefined` keys никогда не совпадают. Несколько matches сохраняют стандартную relational cardinality и создают runtime warning. `join` без префикса остаётся string operation и не используется для relational collections.

```ts
fullJoin(arrivalPairs, departurePairs)
  .byAny('arrivalLeg.id', 'departureLeg.id')
  .coalesce()
```

Object projection DataView вычисляется один раз над целым input:

```ts
defineDataView({
  output: {
    pairs: fullJoin(
      path('pairsArrival'),
      path('pairsDeparture'),
    ).byAny('arrivalLeg.id', 'departureLeg.id').coalesce(),
  },
})
```

В Store `.select({...})` является compiler sugar над таким inline DataView и не создаёт отдельный runtime mechanism.

Selector/predicate descriptors: `get`, `getOr`, `pick`, `omit`, `match`, `eq`, `ne`, `gt`, `gte`, `lt`, `lte`, `includes`, `and`, `or`, `not`, `isNil`, `isEmpty`, `has`. `pick('nested.path')` сохранён как совместимый path selector; для объектной проекции используется `pick(['id', 'name'])`.

Произвольные callbacks, `reduce`, mutation (`set/unset/update`), async и пользовательские comparators намеренно не входят в DSL. Любое выражение компилируется в статический IR и не может выполнить пользовательский JavaScript.

## Runtime host

Runtime host создаётся из valid artifact и владеет live state, Raph paths, subscriptions, timers, abort controllers и child hosts. Уничтожение host обязано быть идемпотентным и освобождать все эти ресурсы.

- `FilterRuntimeHost` владеет state, defaults, outputs и persistence.
- `QueryRuntimeHost` владеет props, запросом, ordered outputs и latest-wins lifecycle.
- `CompositionRuntimeHost` владеет графом child hosts, bindings и hooks.
- `FilterViewRuntimeHost` описывает renderable-проекцию одного `FilterRuntimeHost`, но не владеет вторым Filter state.
- Component/Table/SFC hosts владеют runtime input, но rendering выполняется адаптером.

## Composition

`Composition` — runtime graph и orchestration. Она объявляет Store/Vocab-зависимости в `data`, создаёт Filter, Query, FilterView, Component и вложенные Composition hosts, связывает props через outputs/data и запускает Query hooks. Composition не содержит layout, DOM, Vue config и никогда не имеет capability `renderable`.

Control dependency Query задаётся hooks и компилируется в `CompositionRuntimeGraph`: `onMount().run(target)` создаёт root-run, `onChange('runtime.output').run(target)` реагирует на изменение output, `onChange(prop('path')).run(target)` — на структурное изменение public prop текущей Composition, `onSuccess(source).run(target)` запускает target только после успешного выполнения source Query. Все готовые hooks одного уровня выполняются параллельно; source order не создаёт последовательность. Initial mount ожидает всю success-chain, начатую root hooks. Циклы bindings и hooks запрещены compiler-ом. Runtime host читает только готовые graph edges из `Endge.program` и не интерпретирует source. `.withProps(...)` передаёт значения реактивно, но само по себе не создаёт команду `run`: повторный запрос всегда остаётся явной control dependency.

`store(identity)` является contextual dependency: runtime сначала применяет explicit `dataRuntimes`/`.withData(...)`, затем ищет ближайший Store provider с той же identity и slot только среди Composition ancestors, а при отсутствии создаёт локальный owned fallback. Поэтому одна Composition одинаково запускается внутри project tree и самостоятельно в preview. Alias остаётся локальным именем для `fromData`, provider ищется по Store identity и optional `.slot(name)`. `.isolated()` всегда создаёт локальный instance, `.injected()` запрещает fallback. Borrower никогда не уничтожает provider; sibling fallbacks не видят друг друга и не зависят от mount order.

Вложенная композиция объявляется как `composition('identity')`, монтируется вместе со своим runtime graph и предоставляет родителю только публичные outputs. Вызываемая Composition объявляет public contract через `props: defineProps({...})`, читает его через `prop(path)`, а caller явно передаёт значения через `.withProps({...})`. Это тот же author-facing vocabulary, который используется у Query и Component; различия между runtime hosts остаются внутренней реализацией. Родитель читает объект всех outputs через `fromOutput('runtime')` или один output через `fromOutput('runtime', 'output')`, передаёт Store instances через `.withData({ childAlias: data('parentAlias') })` или публикует outputs через `.storeTo(data(...), mapping)`. Уже рассчитанные к моменту монтирования публичные outputs публикуются начальным атомарным batch, дальнейшие изменения — реактивно. Lifecycle дочернего дерева принадлежит родительскому `CompositionRuntimeHost`. Прямые и транзитивные циклы запрещены.

```ts
defineComposition({
  props: defineProps({
    requirements: field('Object'),
  }),

  runtimes: {
    attributes: query('attributes-leg-select').withProps({
      names: prop('requirements.arrival.attributes'),
    }),
  },

  hooks: [
    onChange(prop('requirements.arrival.attributes'))
      .debounce(200)
      .run('attributes'),
  ],
})
```

Caller может передать literal, обычный Composition binding или compiled metadata. `metadataOf(runtime)` разрешает identity через runtime alias и возвращает весь `ProgramArtifact.metadata.self`; `metadataOf(runtime, namespace)` выбирает `ProgramArtifact.metadata.self[namespace]`. Обе формы доступны внутри ValueExpression и не читают SFC source или renderer state:

```ts
composition('groundhandling-default').withProps({
  requirements: metadataOf('table'),
})
```

Одноаргументная форма сохраняет корневую структуру metadata и не unwrap-ит автоматически единственный namespace.

Для standalone preview Composition может объявить `previewProps: definePreviewProps({...})`. Ключи обязаны совпадать с public props; значение является static JSON literal или explicit `mock(identity)` reference. Compiler хранит fixtures отдельно от props contract, индексирует RMock dependency и публикует preview/type проблемы только как warnings. Configurator materializes fixtures и передаёт их в обычный mount `{ props }`. Production mount, вложенная Composition и field defaults никогда не читают `previewProps` автоматически.

```ts
defineComposition({
  props: defineProps({
    requirements: field('GroundHandlingQueryRequirements'),
  }),

  previewProps: definePreviewProps({
    requirements: mock('groundhandling-query-requirements'),
  }),

  runtimes: {},
})
```

### Composition scopes и resources

Каждая Composition получает неявный `scope_default`. Root resources, direct runtimes и named scopes принадлежат ему. Named scope задаёт lifecycle ownership, но не является persisted entity, `RuntimeHost`, DOM scope или security boundary.

```ts
defineComposition({
  activateOn: startup(),

  resources: {
    applicationTheme: style('groundhandling-themes'),
    common: i18n('application-common-default'),
  },

  runtimes: {
    core: composition('groundhandling-default')
      .activateOn(startup()),

    pages: scope({
      runtimes: {
        control: scope({
          resources: {
            tableTheme: style('groundhandling-control-table'),
          },
          runtimes: {
            content: composition('groundhandling-control-page')
              .activateOn(startup()),
          },
        }).activateOn(manual()),
      },
    }),
  },

  outputs: {
    controlPage: output().fromScope('pages.control'),
  },
})
```

Compiler сохраняет authored activation override отдельно и после dependency linking записывает `effectiveActivation`. Runtime не перечитывает source и применяет порядок `call-site override → root policy child Composition → owner scope policy → startup default`. Call-site override действует только на один instance и не меняет внутренние policies вызываемой Composition.

`i18n(identity)` компилируется в locale indexes внутри Composition artifact. Имя resource является публичным alias функции `t('alias:key', fallback?)`. Catalog накапливается по scope и через вложенные Composition; повтор полного ключа ниже по ancestry является compile error. Runtime не читает глобальный список i18n-документов. Specific override для переводов не входит в этот этап.

`startup()` означает автоматическую активацию относительно непосредственного owner scope. `manual()` создаёт стабильный handle без RuntimeHost. Explicit `Endge.runtime.composition.mount()` является командой activation самой Composition, но не запускает её внутренние manual scopes.

### Lifecycle ownership

`RuntimeScope` владеет child scopes, RuntimeHost membership и `RuntimeResourceBag`. Операции `activate`, `pause`, `resume`, `deactivate`, `dispose` сериализуются одной transition queue. Pause закрывает update gate, отменяет generation signal, останавливает descendants children-first, suspends resources и отмечает отброшенные события как stale. Resume сначала возвращает resources parent-first, затем ранее активные descendants, выполняет один `reconcile()` и только после этого открывает update gate.

Deactivate уничтожает runtime tree через центральный `Endge.runtime` registry и освобождает resources в обратном порядке. Dispose дополнительно удаляет reusable scope handle. `Endge.reset()` ожидает завершения scope disposal, поэтому новый boot не наследует timers, observers, subscriptions или style leases прошлого поколения.

### Project runtime session

Project запускается в одном structural context:

```ts
const project = await Endge.runtime.project.mount('groundhandling-project')
const entry = project.compositions.require('project-entry')
const page = entry.output<RuntimeScopeHandle>('controlPage')

await page.activate()
await page.pause()
await page.resume()
await page.deactivate()
await project.unmount()
```

Runner выбирает Composition documents с `kind: 'project'` и matching `kindIdentity`. Root `startup()` compositions запускаются сразу, `manual()` остаются handles. `switchScope()` проверяет принадлежность одной session, подготавливает target и batch-ит style notifications; при ошибке target исходный scope не выключается.

### Style resources

Compiled `RStyle` только available. Он не становится runtime-active из-за persisted `active=true`. `style(identity)` получает reference-counted lease в boundary owner scope. Pause suspends lease без recompilation, resume возвращает его, deactivate release-ит. DOM materializer prefix-ит output через `data-endge-runtime-scope`, а Canvas может читать тот же neutral active-placement contract.

## Renderer

Renderer получает готовый host с capability `renderable` и не создаёт второй state, persistence или orchestration. `FilterViewRuntimeHost` строит renderer-neutral model полей, а UI adapter выбирает только конкретную реализацию `Input`, `Textarea`, `Checkbox` и `Select`. Query запускается hooks Composition или вызывающим кодом, но не Filter renderer-ом.

## Filter view

Filter и его UI описываются отдельными runtime instances:

```ts
runtimes: {
  filter: filter('schedule'),

  filters: filterView('filter'),

  compactFilters: filterView('filter')
    .fields(['search', 'cancelled'])
    .controls({
      search: control('Input'),
      cancelled: control('Checkbox'),
    })
    .withProps({
      showLabels: true,
      labels: {
        search: 'Поиск рейса',
        cancelled: 'Только отменённые',
      },
    }),
}
```

`.fields(...)` необязателен: без него view отображает все поля Filter. `.controls(...)` тоже необязателен: `vocab` и options дают `Select`, Boolean даёт `Checkbox`, остальные scalar-поля — `Input`. Явный control всегда имеет приоритет над inference.

`.withProps(...)` принимает JSON-compatible literals и стандартные Composition bindings. Props хранятся в `FilterViewRuntimeHost`, попадают в renderer-neutral model и реактивно обновляются для `fromData`, `fromStore`, `fromOutput` и `fromFilter`. Встроенный generator стабильно понимает `showLabels` и `labels`; неизвестные props не передаются в DOM и остаются в model для пользовательского renderer-а.

Один `FilterRuntimeHost` может иметь несколько FilterView hosts с разными наборами полей. Все они читают и меняют единый Filter state через source host. Визуальный adapter не решает, какой control нужен: он получает уже выбранный renderer-neutral control и только рисует его в своей UI-системе.

## Legacy Filter fields и Filter source

`Filter.fields` и `Filter.source` — два независимых persisted-контракта с независимыми consumers:

- legacy UI читает и сохраняет только `fields`;
- source editor читает и сохраняет только `source` и `sourceVersion`;
- compiler/runtime source-first ветки никогда не строят source из `fields`;
- сохранение source никогда не переписывает `fields`;
- пустой source допустим для legacy UI, но даёт compile diagnostic и не может быть смонтирован новым runtime.

Такое сосуществование является осознанным compatibility boundary, а не двусторонней синхронизацией.

## Persistence

Публичный Filter DSL задаёт только стабильный ключ:

```ts
filter('schedule').persist({ key: 'schedule-filter' })
```

Имя поля в `runtimes` является именем runtime instance. Отдельного публичного `.instance(...)` у Composition нет.

Durable storage address строится из `storageId` и полного scope текущего `Endge.context`:

```text
workspaceId + tenantId + projectId + environmentId + userId + persist key
```

`runtimeId` идентифицирует активный host и controller, `storageId` — durable state. Persisted document хранит только Filter state; outputs всегда вычисляются заново.
