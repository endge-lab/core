# Стандарт frontend-архитектуры

## Главный принцип

Пишем максимально простой и сильный код без лишних слоев, абстракций и усложнений.

Правило простоты:

- если задачу можно решить меньшим количеством сущностей без потери читаемости и тестируемости, выбираем более простой вариант;
- не добавляем слой, класс, helper, facade, config или generic «на будущее», если он не дает явной пользы уже сейчас;
- не поддерживаем legacy-паттерны только потому, что они уже существуют;
- не смешиваем простоту с примитивностью: код должен оставаться строгим, явным и удобным для изменения.

## Базовая архитектура

Frontend-фича строится по FSD и элементам чистой архитектуры:

- `ui` — только `.vue`-компоненты;
- `ui/pages` — страницы фичи;
- `ui/modals` — модальные окна фичи;
- `ui/components` — общие переиспользуемые компоненты фичи;
- `ui/module/*` — крупные UI-блоки или подгруппы фичи;
- `domain` — доменные сущности и типы фичи;
- `domain/types` — типы, интерфейсы, DTO и контракты, если их несколько;
- `model` — модули, фасады, сервисы, orchestration-слой;
- `model/config` — конфиг и расширяемые константы фичи;
- `mock` — мок-данные и mock-сервисы фичи;
- `tools` — утилиты фичи;
- `shared/tools` — утилиты, общие для всего приложения.

Каждая фича должна иметь явную точку входа через `index.ts` и наружу экспортировать только свой публичный API.

## Модули и состояние

Стандартный вход в stateful-фичу — модуль федерации (`*_Module`, `*_Federation`).

Правила:

- модуль является source of truth по состоянию фичи;
- только модуль меняет состояние;
- сервисы не мутируют состояние;
- UI не мутирует состояние напрямую;
- если нужен application/presentation-слой между модулем и Vue, используем `Facade`;
- модуль должен иметь понятный lifecycle: `setup`, `init`, `reset`, при необходимости `destroy`.

`composable` допускается только как тонкий Vue-adapter для:

- lifecycle bridge;
- DOM, scroll, focus, gesture, viewport и других browser-specific сценариев;
- связывания Vue с модулем или фасадом.

Бизнес-логика, состояние фичи, orchestration data-flow и mapping domain -> feature-state не должны жить в composable по умолчанию.

## Facade-слой

`Facade` нужен, когда:

- есть derived state для страницы или группы компонентов;
- нужен mapping server/domain records -> UI model;
- одна страница координирует несколько модулей или сервисов;
- нужно скрыть от `ui` внутренние детали модуля.

`Facade` не должен подменять модуль и не должен напрямую владеть источником истины, если state уже живет в модуле.

## Типы и интерфейсы

Типы и интерфейсы не храним в файлах модулей, сервисов, фасадов и других class-файлах, кроме совсем локальных случаев.

Правила:

- если тип используется в двух и более файлах, выносим его;
- если тип является частью контракта слоя, выносим его;
- если типов несколько, используем `domain/types`;
- файлы типов именуем как `*.type.ts`;
- если файл один и он небольшой, допускается хранить его в корне `domain`.

## Сервисный слой

Получение данных — отдельный слой.

Правила:

- API-вызовы инкапсулируются в сервисах;
- сервис работает с сетью, storage, transport и raw payload, но не управляет состоянием страницы;
- модуль получает данные из сервиса и уже затем меняет свое состояние;
- для сервисов, завязанных на runtime-источник данных, задаем явный контракт;
- если фича должна работать без сервера или с демо-данными, добавляем минимум две реализации: `ApiService` и `MockService`.

Для моков:

- сырые mock payload держим в `mock/<feature>`;
- mock-данные делим по файлам и сценариям, а не складываем всё в один гигантский объект;
- mock-реализация должна повторять внешний контракт сервиса.

## Конфиг и константы

Глобальные и расширяемые константы не держим внутри модулей и сервисов, если это не совсем локальная изолированная логика.

Правила:

- конфиг фичи хранится в `model/config`;
- runtime-переключатели, лимиты, timeout, feature flags, defaults и strategy-константы выносятся в config;
- мелкие приватные константы внутри класса допустимы только если они не являются частью фичевого контракта.

## Утилиты

Если утилита относится только к одному классу, оформляем ее как приватный метод этого класса.

Правила:

- все приватные поля, методы и свойства начинаются с нижнего подчеркивания;
- если утилита нужна нескольким сущностям одной фичи, выносим в `tools` фичи;
- если утилита нужна нескольким фичам, выносим в `shared/tools`.

Не плодим «общие helper-файлы» без ясной зоны ответственности.

## Dependency Injection

Классы должны получать зависимости явно.

Правила:

- модуль принимает сервисы, конфиг, логгер и другие внешние зависимости через конструктор или фабрику;
- не создаем сеть зависимостей глубоко внутри метода без необходимости;
- переключение между `ApiService` и `MockService` должно происходить в одном понятном месте bootstrap/config-слоя.

## Именование файлов

Правила именования:

- сущности без явного layer-смысла — `CamelCase.ts`;
- Vue-компоненты с блочной логикой — `*_Button.vue`, `*_Dialog.vue`, `*_Modal.vue`, `*_Page.vue`;
- классы слоя — `*_Service.ts`, `*_MockService.ts`, `*_Module.ts`, `*_Federation.ts`, `*_Facade.ts`, `*_Helper.ts`;
- типы — `*.type.ts`.

Именование должно сразу отвечать на вопрос, что это за сущность и к какому слою она относится.

## UI и стили

Приоритет для UI:

- сначала используем возможности PrimeVue и его токены;
- затем добавляем фичевый CSS/SCSS только там, где компонентная библиотека не покрывает сценарий;
- большие блоки стилей не держим внутри `.vue`, кроме локально изолированных случаев.

Правила:

- основные стили храним в `assets`;
- для общих стилей приложения используем единую точку импорта вроде `main.scss` или `app.scss`;
- компонент может подключать отдельный изолированный файл стилей напрямую, если стиль принадлежит только ему;
- визуальный рефакторинг не должен ломать существующий layout, адаптив и поведение.

## Комментарии и читаемость

Пишем короткие русские комментарии в формате JsDoc там, где они реально повышают читаемость.

Правила:

- обязательно комментируем публичные методы классов;
- обязательно комментируем нетривиальные приватные методы;
- не комментируем очевидный код ради формальности;
- между логическими блоками кода оставляем пустые строки.

Формат:

```ts
/**
 * Краткое объяснение назначения.
 */
```

## Границы слоев

Нельзя смешивать слои.

Правила:

- в `ui` не держим transport DTO, network helpers, config и сервисную инициализацию;
- в `model` не держим DOM-логику и разметочные детали;
- `domain` не зависит от Vue, браузерного API и transport-слоя;
- mapping выполняем на границах слоя, а не размазываем по всему коду.

## Чек-лист перед merge

Перед merge проверяем:

- код стал проще, а не сложнее;
- модуль или фасад явно владеет состоянием;
- типы и интерфейсы не прячутся в class-файлах без причины;
- data-flow отделен от UI;
- мок-слой и API-слой не смешаны;
- константы вынесены в config, если они не локальные;
- стили не распухают внутри `.vue` без необходимости;
- структура фичи читается по папкам и именам файлов;
- визуал и логика не деградировали.
